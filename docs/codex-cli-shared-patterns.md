# Codex Companion CLI 共性模式

这份文档总结 `codexs` 与 `codexm` 当前已经验证有效的共同结构、交互模式和实现边界，目标不是描述某一个命令，而是提炼一套可复用的 Codex companion CLI 骨架，供后续 `codex-search`、`codex-team` 以及更多同类工具复用。

## 1. 共同定位

`codexs` 和 `codexm` 都不是 Codex 的执行内核，而是 Codex 外围的入口层、路由层和治理层。

| 维度 | `codexs` | `codexm` | 共性 |
| --- | --- | --- | --- |
| 主问题 | 找到并重新进入正确 thread | 让实例运行在正确的 managed runtime | 都在降低“回到工作现场”的成本 |
| 主要输入 | 本地 thread 历史、状态库、用户查询 | 本地 auth 快照、quota、Desktop / CLI 运行态 | 都强依赖本机状态 |
| 主要动作 | 搜索、浏览、打开、resume | 查看、切换、launch、watch、run | 都是“读本地状态，再触发本地动作” |
| 面向对象 | 人类交互和脚本调用 | 人类交互和脚本调用 | 都必须同时兼顾可读性和可编排性 |
| 核心价值 | 更快回到上下文 | 更稳定地进入可治理实例 | 都是在减少中断和认知负担 |

一个更准确的抽象是：

1. 感知当前上下文
2. 选择合适入口
3. 执行本地动作
4. 在不满足条件时降级
5. 把过程变成可观测事件

## 2. 共同产品骨架

两者已经收敛出同一种 companion CLI 工作流：

| 阶段 | `codexs` | `codexm` | 复用结论 |
| --- | --- | --- | --- |
| Sense | 读取 sessions、title、active / archived 状态 | 读取 auth、quota、Desktop / CLI runtime | 先做本地状态感知，再做决定 |
| Decide | 选择 `open`、`resume`、`lucky` | 选择 `current`、`switch`、`launch`、`run`、`watch` | 决策层应独立于执行层 |
| Act | 打开 deeplink 或执行 `codex resume` | 切换 auth、启动 Desktop、包装 CLI、启动 watcher | 平台动作要收口到单点 |
| Degrade | archived 只读、非 TTY 强制 `--json` / `--jsonl` | Linux / WSL 下 Desktop 命令降级到 CLI 路径 | 必须显式表达无法满足的前提，而不是静默失败 |
| Observe | 搜索元日志、事件日志、TUI 状态 | watch 日志、quota 历史、debug 流 | 每条关键路径都应可诊断 |

这套骨架的重点不是“命令多”，而是“每一步的边界清晰”。

## 3. 共同代码结构

两边都适合用“薄入口 + spec 驱动 CLI + 稳定职责领域层”的组织方式。

| 层级 | `codexs` | `codexm` | 复用结论 |
| --- | --- | --- | --- |
| 入口编排 | `src/main.ts` | `src/main.ts` | `main` 只做命令分发、依赖拼装和统一错误出口 |
| CLI 规格 | `src/cli/spec.json` + `spec.ts` | 同类 `spec.json` + `spec.ts` 结构 | 帮助、参数、补全应从同一份 spec 派生 |
| 参数解析 | `src/cli/args.ts` | `src/cli/args.ts` | 参数校验和业务逻辑分离 |
| Help / Completion | `src/cli/help.ts` | `src/cli/help.ts` | help 是产品面，不是附属品 |
| 输出格式化 | `src/cli/output.ts` | `src/cli/quota.ts` 等展示模块 | 人类视图和机器视图分轨 |
| 领域模块 | `src/search/*`、`src/tui/*` | `src/account-store/*`、`src/watch/*`、`src/desktop/*` | 领域逻辑按稳定职责拆，不按命令名横切 |

可继续复用的关键原则：

- `spec` 是单一事实源，避免 help、README、completion 和实现漂移。
- 平台相关能力集中到 adapter / launcher / runner 一类模块。
- 领域层表达意图，不散落系统命令、路径差异和平台分支。

## 4. 共同 UI / UX 模式

### 4.1 先给快路径，再给完整控制

两者的默认体验都不是“先讲清所有选项”，而是优先让用户走到最常见成功路径。

- `codexs <keyword>` 默认直接进入 TUI 搜索。
- `codexs lucky <keyword>` 直接走最新可打开 thread。
- `codexm current`、`codexm list` 默认给当前最关心的状态概览。
- `codexm run` 和 `codexm watch` 分别承接 CLI 连续运行和托管监控的快路径。

### 4.2 人类界面与机器界面并存

两者都不是纯脚本工具，也不是纯交互工具，所以默认需要双轨输出：

- 默认输出优化人读和决策。
- `--json` 提供稳定机器消费。
- 流式场景优先 `--jsonl`、持续刷新列表或 watcher 输出，而不是等全量结束。

### 4.3 流式优先，而不是全量等待

这是两者最稳定的交互收益来源之一。

- `codexs` 会先显示首批命中的 thread，再持续回填更多 match。
- `codexm watch`、`codexm run` 都属于持续观察和连续运行场景，本质上是增量输出而不是一次性结果。

可复用原则：

- 首个可操作结果优先于全量精确结果。
- “还在做事”必须有明确状态。
- 长路径动作要让用户感知到阶段变化，而不是无反馈阻塞。

### 4.4 全局状态与局部对象分层

`codexs` 的 footer 与 detail header 已经证明，这是一条很值得复用的规律；`codexm` 的概览表与 watcher 行输出也遵循同样思路。

- 全局区域只放全局进度、范围、模式、计数、连接状态。
- 当前对象的私有信息放详情区、行内或对象级输出。
- 默认列表或概览先给“做决策必须的信息”，更细上下文按需展开。

### 4.5 托管是增强，不是强制打断

这是后续同类工具很重要的 UX 原则。

- 能进入 managed 路径时优先进入。
- 不能进入时明确降级原因。
- 已有实例在跑时，优先避免无必要接管、重启或打断。
- “best effort managed” 比“表面强制托管、实际副作用不可控”更稳。

### 4.6 TUI 必须响应终端尺寸，而不是只适配默认窗口

这是 companion CLI 很容易被忽略、但对真实使用体验影响很大的约束。

- TUI 布局必须同时考虑宽度和高度，不能只按“宽屏/窄屏”二分。
- 详情区、列表列数、底部操作提示都应随可用空间渐进退化，而不是在小终端里硬塞完整信息。
- 终端变窄、变矮时，优先保留：当前可操作对象、关键状态、核心动作；次级元信息和长 preview 应让位。
- 终端变宽、变高时，再恢复更完整的 detail metadata、preview 和快捷键说明。
- 布局判断不仅要在首次渲染时生效，也要对运行中的 resize 事件做即时响应。

一个值得复用的默认顺序是：

1. 顶部保留单行全局摘要
2. 中间区域优先保证列表可操作
3. 底部固定为交互 dock：输入 / 状态 / 快捷键提示
4. 详情区按空间做渐进披露，而不是和列表抢最小可用空间

### 4.7 底部交互区要模式化，而不是只放状态消息

`codexs` 和 `codexm` 当前都证明了一个稳定模式：底部不是单一 footer，而是承接“输入 + 状态 + 动作提示”的交互区域。

- 搜索、筛选、确认、review 等高注意力输入优先靠近底部。
- 状态行应明确表达当前模式、正在进行的动作和失败原因，而不是只显示静态摘要。
- 快捷键提示应按当前模式切换，并按宽度做优先级裁剪。
- 小终端里宁可保留高频动作的短提示，也不要把所有按键平铺成噪声。

### 4.8 可以复用的一组 TUI UI / UX 优秀实践

如果后续继续打磨 `codexs`、`codexm`、`codext` 或同类工具，下面这些点值得当作默认风格约束。

#### 4.8.1 顶部只放全局摘要，不塞对象细节

- 顶部优先放：当前模式、tab、计数、过滤范围、更新时间、全局健康状态。
- 顶部不要重复详情区的私有字段，例如完整路径、长 preview、对象级诊断原因。
- 顶部最好压成单行；如果必须两行，第二行也应是全局摘要，不应变成“另一个详情区”。

#### 4.8.2 列表只展示做决策必需的信息

- 列表列应该优先服务“快速选择”，而不是信息最全。
- 默认列建议只保留：选择态、主标题、对象归属、最近时间或核心排序字段。
- 宽度不足时，优先裁掉次级列，不先裁主标题和焦点高亮。
- 焦点态、选中态、当前对象态应使用稳定且可一眼识别的视觉标记。

#### 4.8.3 详情区负责补足上下文，不负责抢主操作路径

- 详情区适合放：`id`、生命周期、路径、元数据、诊断原因、preview。
- 元信息展示顺序应固定，优先级通常是：
  1. 身份与生命周期
  2. 可执行动作所需的定位信息
  3. 补充上下文和 preview
- 高度不足时，应优先保留 `id / status / lifecycle / path`，再决定是否展示长 preview。

#### 4.8.4 底部快捷键说明必须是“模式化提示”，不是静态帮助文本

- list 模式、search/filter 模式、confirm 模式、review 模式都应有不同的快捷键提示。
- 快捷键提示应按优先级退化：
  1. 当前模式下最常用、最关键的动作
  2. 次一级导航动作
  3. 低频补充动作
- 宽终端显示完整提示，窄终端显示高优先级短提示。
- confirm / destructive 模式下，提示应明确 `confirm` 和 `cancel` 键位，而不是让用户猜。

#### 4.8.5 交互键位应在不同工具之间保持稳定词汇

下面这组词汇已经在 `codexs` / `codexm` 中证明比较稳，后续最好尽量复用：

- `j/k`：上下移动
- `Enter`：当前模式下的主动作
- `Space`：展开、选择或 toggle
- `Tab`：切换 tab / focus / 子区域
- `/`：进入 search 或 filter
- `Esc`：返回、取消、退出当前临时模式
- `q`：退出整个 TUI

这能显著降低用户在不同 companion CLI 之间切换时的重新学习成本。

#### 4.8.6 状态文案要表达阶段，不只表达结果

- 长动作不应只在结束后报结果；开始、进行中、等待确认、失败都应有明确状态。
- 状态行优先回答：
  - 当前在做什么
  - 是否还能继续操作
  - 如果失败，失败发生在哪一步
- 错误状态和正常摘要不要混在同一层级里竞争注意力。

#### 4.8.7 小终端的目标不是“展示完整”，而是“保持可操作”

- 极窄或极矮终端的首要目标，是保证用户还能移动、筛选、确认和退出。
- 当空间不足时，应优先保留列表和底部操作 dock，而不是强留完整详情区。
- “列表优先 + 状态清楚 + 动作清楚” 通常比“详情完整但已经难以操作”更对。

## 5. 共同可观测性模型

可观测性不是附加项，而是 companion CLI 的基础设施层。没有它，就很难判断问题出在搜索、路由、托管、平台调用还是降级逻辑。

### 5.1 为什么它是第一等能力

可观测性至少直接服务三类目标：

| 目标 | 需要回答的问题 |
| --- | --- |
| 性能优化 | 慢在搜索、状态读取、Desktop 唤起、CLI resume，还是 quota 轮询？ |
| 功能优化 | 用户最常走哪条路径？managed 命中率如何？fallback 多不多？ |
| 功能排障 | 失败发生在哪一步？平台条件是什么？是否已降级？为什么降级？ |

### 5.2 两边已经存在的三层形态

当前 `codexs` 和 `codexm` 实际上已经凑齐了统一可观测性的三个层次。

| 层次 | `codexs` | `codexm` | 共享意义 |
| --- | --- | --- | --- |
| 事件流 | `events.jsonl` 记录搜索、打开、resume、失败事件 | watcher 状态变化、auto-switch、run 重启、连接丢失 / 恢复 | 复盘调用链路 |
| 元数据日志 | `searches.jsonl` 记录 query、flags、duration、results、status | `watch.log`、debug 输出记录运行阶段和异常 | 定位阶段性行为和失败原因 |
| 历史样本 | 搜索完成态和文件扫描进度 | `watch-quota-history.jsonl` 记录 quota 观测样本 | 支撑趋势分析、排序、ETA 和诊断 |

这三层分别回答不同问题：

- 事件流回答“发生了什么”。
- 元数据日志回答“这次调用是怎么走完的”。
- 历史样本回答“长期看系统怎么变化”。

### 5.3 统一设计建议

如果以后继续做第三个同类工具，建议默认按下面的规则设计：

- 关键路径都产出结构化事件，而不是只写人类日志。
- 记录元信息和链路，不默认记录 thread 内容、prompt 内容或敏感账号数据。
- 每个关键动作至少带上：`tool`、`action`、`mode`、`result`、`duration_ms`、`error_code`、`fallback_reason`。
- 日志写入必须是 best effort，不能反过来影响主流程。
- 事件命名优先稳定语义，如 `search_started`、`managed_route_selected`、`fallback_used`、`operation_failed`。

## 6. 对后续 CLI 最值得复用的能力

如果后续再做更多 Codex companion CLI，优先抽象这些能力：

| 可抽象能力 | 当前在 `codexs` / `codexm` 的体现 | 抽象价值 |
| --- | --- | --- |
| spec 驱动 help / usage / completion | 两边都有 | 最高 |
| 参数 / 命令 suggestion | 两边都有 | 高 |
| 人类输出与机器输出双轨 | 两边都有 | 高 |
| 流式结果与持续反馈 | 搜索流、watch / run 输出 | 高 |
| 平台动作 adapter / launcher / runner | `open` / `resume`、Desktop / CLI 托管 | 高 |
| 降级语义 | archived 只读、平台降级、托管失败回退 | 高 |
| 结构化事件与元日志 | `searches.jsonl`、`events.jsonl`、`watch.log`、quota history | 高 |
| 全局状态与对象详情分层 | TUI footer/detail、概览表/watcher 行 | 中高 |

## 7. 结论

`codexs` 和 `codexm` 虽然一个偏 thread 搜索，一个偏账号与运行态治理，但已经收敛出了同一种 Codex companion CLI 形态：

- 用 spec 统一命令面。
- 用薄入口编排复杂动作。
- 用流式和增量更新提升交互速度。
- 用全局状态与局部详情分层控制信息密度。
- 用显式降级保持行为可预期。
- 用结构化可观测性支撑性能优化、功能优化和问题排查。

如果后续继续做同类工具，建议默认从这套骨架出发，而不是每次重新发明 CLI 结构。
